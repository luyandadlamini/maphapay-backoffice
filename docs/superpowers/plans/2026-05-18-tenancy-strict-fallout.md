# Tenancy Strict-Mode Fallout

Generated by Task 1.4 of the Money Movement Hardening plan (see [the parent plan](./2026-05-18-money-movement-hardening.md)).

## Test suite status

The `UsesTenantConnection` trait short-circuits in `app.env === 'testing'` (returns `null` = default connection). Because the Pest suite always runs with `app.env=testing`, the strict path is **not** exercised by ordinary test runs. The contract test in `tests/Feature/Architecture/TenantBoundaryEnforcementTest.php` explicitly flips `app.env` to `'local'` to cover it.

Implication: the dynamic test-driven audit envisioned in the original plan (Phase 1 Step 5 "the failing test suite under strict mode is the inventory") does not apply as written. A **static audit** is required instead.

## Static audit needed (Task 1.5)

For each non-HTTP entry point, verify that any read or write to a `UsesTenantConnection` model is wrapped in `WithTenantContext::withAccountTenancy($accountUuid, fn() => ...)`. Likely failure surfaces, in priority order (highest-impact first for money movement):

1. **Projectors** under `app/Domain/**/Projectors/*.php` — these run in queue workers with no tenancy middleware.
   - Confirmed broken in this plan: `app/Domain/Account/Projectors/AssetBalanceProjector.php` (queries `AccountBalance::firstOrCreate(...)` and `Account::where(...)` without tenancy). Fixed in Phase 3 (Tasks 3.2–3.4) by migrating to `TenantAwareProjector`.
   - Audit pattern: `grep -rln "extends Projector" app/Domain | xargs -I{} grep -L "TenantAwareProjector\|withAccountTenancy" {}`

2. **Reactors** under `app/Domain/**/Reactors/*.php` — same risk profile as projectors.

3. **Temporal activities** under `app/Domain/**/Activities/*.php` — workflow activities execute outside any HTTP context. Each must initialize tenancy from the workflow input.

4. **Queue jobs** under `app/Domain/**/Jobs/*.php` and `app/Jobs/*.php` — same.

5. **Console commands** under `app/Console/Commands/*.php` — CLI invocations have no middleware. Sweep tasks, repair scripts, etc.

6. **Filament admin** — already substantially audited (`WithAccountTenancy` concern added in commit `932decf1`), but spot-check any Filament page that loads a `UsesTenantConnection` model without initializing tenancy via the concern.

7. **Seeders** under `database/seeders/*.php` — if any seeder writes tenant-scoped data, it must wrap in tenancy. For local dev seed data this may be acceptable to leave unfixed (loud failure surfaces real bugs), but for production seed paths fix immediately.

## Audit procedure (Task 1.5)

```bash
# Enumerate candidates per surface, then for each file, manually verify that
# every tenant-model interaction is inside a withAccountTenancy() closure.

# Projectors
grep -rln "extends Projector\|extends \\\\Spatie\\\\EventSourcing" app/Domain

# Reactors
grep -rln "extends Reactor\|extends \\\\Spatie\\\\EventSourcing.*Reactors" app/Domain

# Temporal activities
grep -rln "use.*Temporal\\\\Activity" app/Domain

# Queue jobs touching tenant models
grep -rln "ShouldQueue" app/ | xargs grep -L "withAccountTenancy"

# Console commands
ls app/Console/Commands/
```

For each candidate, the fix is one of:
- Migrate to `TenantAwareProjector` / `TenantAwareReactor` (preferred — automatic).
- Wrap explicitly: `(new class { use WithTenantContext; })->withAccountTenancy($uuid, fn() => /* ... */)`.
- Document in the test allowlist if the call site is genuinely tenant-agnostic.
